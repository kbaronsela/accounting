# ארכיטקטורה — פлатפורמה לשיתוף קבלות/חשבוניות (PWA)

מסמך מאגד לפיתוח MVP. עדכונים עתידיים: גרסה + תאריך בכותרת משנה.

## 1. מטרות MVP

- **לקוח** מעלה או מצלם קבלה/חשבונית, המערכת בודקת קריאות, מנסה לחלץ שדות (OCR/Document AI), הלקוח עורך ומאשר, ושולח לשרת.
- **רואה חשבון** רואה מסמכים לפי לקוחות שלו, מסנן (תאריכים, סכומים וכו׳), ומקבל **התראה במייל** כשמסמך נשלח.
- **אדמין** נוצר ב**פריסה ראשונה** (bootstrap), ויוצר רואי חשבון דרך הזמנות.
- **שפות ממשק**: עברית ואנגלית (כולל RTL לעברית).
- **אין רישום ציבורי**: לקוחות ורואי חשבון נכנסים רק דרך **הזמנה** עם טוקן.

## 2. סטאק טכנולוגי (החלטות)

| שכבה | בחירה |
|------|--------|
| אפליקציה | PWA (דפדפן + התקנה למסך הבית), מומלץ **Next.js (App Router)** + TypeScript |
| אימות | **Auth.js** (NextAuth v5) — סיסמה + ספקי OAuth (למשל Google, Facebook) לפי הגדרה |
| מסד נתונים | **PostgreSQL** |
| אחסון קבצים | Object storage סביר (S3 / R2 וכו׳) עם העלאה חתומה (presigned) |
| OCR | שירות Document AI בענן (נבחר בפיתוח; לדוגמה Google/Azure/AWS) — אסינכרוני |
| מייל טרנזקציוני | ספק חיצוני (Resend / Postmark / SendGrid וכו׳) |

## 3. תפקידים והרשאות

| תפקיד | תיאור קצר |
|--------|------------|
| `admin` | נוצר ב-bootstrap; יוצר רואי חשבון ומזמין אותם |
| `accountant` | מנהל לקוחות (תיקים), מזמין משתמשים לתיק, צופה במסמכים ומסנן |
| `client` | חבר ב-`client_members`; מעלה מסמכים לתיק אליו הוא משויך |

משתמש אחד יכול לקבל **מספר תפקידים במקביל** (רשומות בטבלת `user_role`), למשל `admin` + `accountant`.

**אין** רישום עצמאי ללקוח או לרואה חשבון — רק דרך `Invitation`.

## 4. זרימות Onboarding

### 4.1 אדמין ראשון

- נוצר ב**פריסה ראשונה** (סקריפט / miġration / משתנה סביבה לסיסמה זמנית).
- לאחר התחברות ראשונה — מומלץ **חובת החלפת סיסמה** או מדיניות סיסמה חזקה.

### 4.2 רואה חשבון

1. `admin` יוצר רשומת רואה חשבון ומפעיל **הזמנה** (מייל + link עם טוקן חד-פעמי, תוקף מוגבל).
2. רואה החשבון משלים הרשמה (סיסמה ו/או OAuth).
3. **OAuth**: המייל שחוזר מהספק **חייב** להתאים למייל שהוזמן.

### 4.3 לקוח (תיק לקוח)

1. רואה החשבון יוצר **תיק לקוח** (`clients`) ושולח הזמנה לחבר ראשון.
2. משתמש שמקבל הזמנה מסיים הרשמה ונוצר `User` עם תפקיד `client` + שורה ב-`client_members`.
3. **הוספת משתמש נוסף לאותו תיק** (למשל בן/בת זוג): **רק רואה החשבון** — הזמנה נוספת **עם אותו `client_id`**.
4. לקוח לא יכול “להזמין” משתמשים בעצמו.

## 5. מודל נתונים (PostgreSQL) — גרסת MVP

### 5.1 `user` (משתמש — Auth.js + אפליקציה)

- `id`, `email` (ייחודי), `passwordHash` (nullable אם רק OAuth), `locale`, `lastDocumentsSeenAt` (אופציונלי — שמור לשימוש עתידי)
- **אין** עמודת `role` בודדת — ראו `user_role`.

### 5.1.1 `user_role` (תפקידים — כמה לכל משתמש)

- `(userId, role)` מורכב כ-primary key; `role` ∈ `admin` | `accountant` | `client`
- משתמש אחד יכול להחזיק **מספר תפקידים** (למשל גם אדמין וגם רואה חשבון).

### 5.2 `clients` (תיק לקוח אצל רואה חשבון)

- `id`, `accountant_id` → `users` (רק accountant)
- `display_name`, `status` (`pending_invite` | `active` | `archived`)
- `invited_email` (אופציונלי — עוזר לטרום-הרשמה)
- **אין** `user_id` יחיד “בעל התיק”; השיוך דרך `client_members`.

### 5.3 `client_members`

- `client_id`, `user_id`, `role` (`primary` | `member` — אופציונלי)
- אילוץ: `unique(client_id, user_id)`

### 5.4 `invitations`

- `email`, `role` (`accountant` | `client`)
- `token_hash` (לא לשמור טוקן גולמי), `expires_at`, `consumed_at` (nullable)
- `created_by_user_id`
- `client_id` — חובה כש-`role = client` (מצביע על התיק שאליו המוזמן יצטרף)

### 5.5 `documents`

- `client_id`, `uploaded_by_user_id`
- `storage_object_key`, `mime_type`, `byte_size`
- `status` — ראו סעיף 6
- `quality_notes` (טקסט קצר למשתמש)
- `ocr_provider`, `ocr_job_id` (אופציונלי, לניפוי באגים)
- `extracted` (JSON — פלט OCR + confidence אם קיים)
- שדות מנורמלים אופציונליים לשאילתות: `extracted_*` / `final_*` (סכום, מטבע, תאריך, ספק, **מספר חשבונית/קבלה**)
- `client_note` (טקסט חופשי)
- `submitted_at` (nullable עד submit סופי)
- חותמות זמן

### 5.6 `audit_events` (מומלץ)

- `actor_user_id`, `action`, `entity_type`, `entity_id`, `payload_json`, `created_at`

## 6. מכונת מצבים — `Document.status`

מצבים לפי MVP (ניתן לאחד חלקם ביישום):

| מצב | תיאור |
|-----|--------|
| `draft_uploading` | נפתחה בקשה; ממתין לקובץ באחסון |
| `uploaded` | קיימת במאגר; לפני/במהלך בדיקת איכות |
| `quality_check_failed` | אינו קריא מספיק — נדרש צילום/העלאה מחדש |
| `ocr_processing` | OCR רץ (או בתור) |
| `ocr_failed` | כשל טכני בשירות OCR (לא בהכרח “לא קריא”) |
| `needs_review` | יש תוצאות; הלקוח עורך לפני שליחה |
| `submitted` | נשלח לרואה החשבון; מתעדכן `submitted_at` ומתבצעת שליחת **מייל** לרו״ח |
| `approved` | אושר על ידי רואה החשבון בלבד מתוך `submitted`; הלקוח ללא עריכה |
| `archived` | בארכיון (לפי השימוש הנוכחי בקוד) |

**כללי מפתח**

- מעבר ל-`submitted` הוא **חד-כיווני** (ללא מדיניות חריגה ב-MVP).
- מעבר ל-`approved` הוא **פרט לרואה החשבון**, רק מתוך `submitted`.
- התראות לרואה החשבון: **רק** לאחר `submitted`.
- `quality_check_failed`: מומלץ רישום **מסמך חדש** להעלאה חוזרת (פשטות), או מדיניות גרסאות מפורטת בהמשך.

## 7. ולידציה לפני Submit (שדות אופציונליים, פורמט כשנמלא)

לפני `POST .../submit` **אין** שדות חובה: הלקוח יכול לשלוח לרואה החשבון גם עם שדות חשבונית ריקים. השדות נשארים **ניתנים לעריכה** עד ההגשה (ובממשק הרו״ח — בהתאם ל־`editableInvoiceFields` אחרי ההגשה).

כללים כאשר משהו כן צוין:

| שדה | חובת מילוי | ולידציה כשנמלא |
|-----|-------------|----------------|
| סכום | לא | מספר תקף (לא ריק בספרות) |
| מטבע | לא בנפרד; אם הסכום ריק גם המטבע נשמר כ־NULL | בשימוש הנוכחי: מטבע קנוני בשקלים בלבד כשיש סכום |
| תאריך חשבונית | לא | `YYYY-MM-DD` אחרי פרסינג מתצוגת `DD.MM.YYYY` |
| שם עסק/ספק | לא | אורך מקסימלי |
| מספר חשבונית / קבלה | לא | OCR מנסה לחלץ ל־`extracted_invoice_number`; הלקוח והרו״ח יכולים לערוך ל־`final_invoice_number` |
| הערת לקוח | לא | — |

Confidence נמוך אחרי OCR — סימון ויזואלי; אופציונלי דיאלוג אישור לפני ההגשה.

## 8. “חדש” אצל רואה החשבון (עתידי)

ב־MVP **אין** בממשק סינון «רק מסמכים חדשים». שדה `users.lastDocumentsSeenAt` נשמר בסכימה לשימוש עתידי אפשרי (למשל סימון «נקרא» או דשבורד).

**אפשרות עתידית:** מסמכים ב־`submitted` עם `submitted_at > last_documents_seen_at`, או טבלת `document_views` לפי `document_id` + `accountant_user_id`.

## 9. API — קווים מנחים (REST)

הטמעה מדויקת (נתיבים) ייקבעו בקוד; העקרונות:

- **הזמנה**: `GET/POST` סביב טוקן הזמנה (אימות + השלמת חשבון).
- **Admin**: יצירת רואה חשבון + הזמנה, רשימה, השבקה (אופציונלי).
- **Accountant**: CRUD לקוחות (תיקים), הוספת `client_members` בהזמנה, רשימת/פילוח מסמכים, הורדה מאובטחת.
- **Client**: בקשת העלאה (presigned), עדכון שדות, `submit`.

כל endpoint עם הרשאות לפי תפקיד ושיוך ל-`accountant_id` / `client_id`.

## 10. Auth.js — עקרונות יישום

- ספקים: `Credentials` (סיסמה) + `Google` / `Facebook` לפי דרישה.
- לאחר OAuth callback: אימות שה-**email מאושר** תואם **email של הזמנה פעילה** (או משתמש קיים המקושר להזמנה) — אחרת דחייה והסבר.
- יצירת סשן רק למשתמש עם `role` ושיוך תקף (`client_members` / accountant פעיל).
- **Admin ראשון**: ללא הזמנה; נטען מ-bootstrap (לא דרך הרשמה ציבורית).

פרטים טכניים (adapter ל-DB, JWT vs database sessions) ייקבעו בשכבת היישום.

## 11. אבטחה ופרטיות (MVP)

- קבצים פרטיים; הורדה רק עם הרשאה ו-URL חתום קצר-מועד.
- הצפנה של מנוחת קבצים בספק הענן (SSE-KMS וכו׳) לפי יכולת הספק.
- לוגים לפעולות אדמין ולשינויים רגישים (`audit_events`).
- מדיניות שמירה/מחיקה — להגדיר עסקית בהמשך.

## 12. הערות PWA + התראות

- **מייל**: ערוץ ההתראה לרו״ח כשמסמך עובר ל־`submitted` (תלוי הגדרת ספק המייל).
- ציפיות real-time: «מייל תוך דקות» מספיק ל-MVP רב הזמן.

## 13. מה לא ב-MVP (מפורש)

- SMS / WhatsApp.
- רישום ציבורי ללא הזמנה.
- קטגוריות הוצאות מתקדמות (אלא אם תתווסף במכוון).
